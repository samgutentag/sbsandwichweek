1. What this is

This repo is the source of truth for the Santa Barbara food week maps (Burger Week, Coffee Week, Sandwich Week, and whatever comes next). Each event gets its own repo, created from this one with GitHub's "Use this template" button, and each event site is a static page on GitHub Pages: an interactive Leaflet map of participating restaurants with menus, filters, directions links, a public stats dashboard, and a lightweight click tracker running on one Cloudflare Worker.

It is not a CMS and there are no accounts, databases, or servers to babysit. The only live infrastructure per event is the Worker (tracking) and two scheduled GitHub Actions, and all three turn themselves off when the event ends.

What's ahead: the vocabulary, the event lifecycle, the two scripted transitions (start and wind-down), a tour of the visitor-facing screens, how tracking works, how code flows from this template to the event repos, and the shared restaurant registry that means you never hunt down an Apple Maps link twice.

2. Words you need first

Template-owned vs event-owned
Every file belongs to exactly one side. Template-owned files (the app code, stats, worker logic, scripts) are copied wholesale from this repo into event repos and must never be edited there. Event-owned files (config.js, data files, wrangler.toml, snapshots) belong to the event and are never touched by a sync. The boundary is written down in template-manifest.json.
THEME / config.js
The single per-event configuration object: name, emoji, dates, timezone, filters, URLs. Values only. The functions that interpret those values live in template-owned event-state.js, so a config file can never drift behaviorally.
Event state
One of off-season, pre-event, during, post-event, computed from the config dates on the event's clock (timeZone, normally America/Los_Angeles), never the viewer's. A wound-down event sets archived: true, which forces off-season permanently.
Skeleton vs full data
data.js is the pre-launch skeleton (restaurants on the map, menus empty). data-<year>.js is the full file with menus, loaded once dataLiveDate arrives (at 12:01 AM). Same restaurant list in both.
The registry
registry/restaurants.json, in this repo only: every venue ever seen across events, with coordinates, contacts, Google and Apple Maps links, and participation history. New events pull from it; concluded events write back to it.
Snapshot
A committed JSON copy of tracking data. Daily snapshots accumulate during the event; the wind-down bakes a final one into tracking-snapshot.js. Snapshots are the permanent record, because the raw analytics rows expire (about 90 days).
The Worker
One Cloudflare Worker per event (workers/track/). Visitors' clicks POST to it; it writes to a Cloudflare Analytics Engine dataset and serves the aggregate numbers the stats pages read.

3. Prerequisites

Everything runs from a Mac with:

brew install imagemagick        # og-image rendering (optional, apply-theme skips if absent)
npm install -g wrangler         # Cloudflare Worker deploys
# plus: python3 and node, both already used by every script

Per event you also need: a domain (the sb<thing>weekmap.com convention), Cloudflare DNS, three GitHub repo secrets (GOOGLE_PLACES_API_KEY, CF_ACCOUNT_ID, CF_API_TOKEN), and two Worker secrets set with wrangler secret put (CF_API_TOKEN, ADMIN_TOKEN). The README's setup guide walks each one.

4. The life of an event

The site changes itself as the config dates pass. You set three dates once; everything else is automatic, including the shutdowns.

Three more things run on this calendar without help. The hours-refresh Action runs daily at 6am PT but only inside the window 3 days before dataLiveDate through 1 day after the end. The snapshot Action runs daily at 7am PT from 1 day before dataLiveDate through 5 days after the end. Both read the dates from config.js at runtime, so the workflow files themselves never change. Manual runs from the Actions tab always work, window or not.

5. Starting an event scripts/

The full timeline with lead times lives in docs/event-runbook.md (and the /foodweek-tickets skill turns it into dated Linear tickets). The short version:

  1. Create the repo from this template, fill config.js identity fields, and set DATASET_NAME (plus the matching dataset binding) in workers/track/wrangler.toml.
  2. Pull the roster. Put one venue name per line in a text file, then 
    python3 scripts/registry.py pull --roster roster.txt --year 2027
    Known venues arrive with coordinates, contacts, and both map links filled. Unknown ones get a stub with a Google Maps search fallback, and fetch-place-ids.py / fetch-details.py / fetch-hours.py fill the gaps from the Places API.
  3. Validate and launch. 
    python3 scripts/start-event.py
    It refuses to pass with placeholder values, mismatched dataset names, missing tag booleans, or areas that aren't in AREA_COLORS, then runs apply-theme.py and prints the short manual list that remains (secrets, wrangler deploy, DNS, Pages).
apply-theme.py is the re-branding step: it rewrites the HTML shells, og-image, CNAME, README badges, and the Worker's event-window vars from config.js. Run it after any config change. It is safe to run repeatedly.

6. The site your visitors see

The map /

Sidebar list plus Leaflet map, one pin per physical location (multi-location restaurants split into "Name (Location)" pins). Visitors can search, filter by area chips, dietary tags, and hours ("Open Now" / "Lunch" / "Dinner", which appear only once hours.json exists), get directions (Apple or Google), call, share, and thumbs-up a restaurant. On phones the restaurant detail opens as a half-height sheet so the map stays visible. Restaurants that participated in a previous year show a returning badge once they hit 2+ years, driven by THEME.firstYearByName.

During the event a live "eyes" layer marks restaurants people are looking at right now: it counts views in the last 10 minutes, and the header chip totals them. On localhost the eyes are simulated so you can develop the feature without traffic.

Stats /stats

Public dashboard, four tabs (Activity, Insights, Trends, Restaurants) with an All time / Pre-event / During / Post-event range filter. The leaderboard score weighs intent: (Directions + Phone) × 3, (Direct links + Shares + Likes) × 2, (Website + Instagram + Views) × 1. While the event is live the numbers come from the Worker; after wind-down they come from the committed snapshots, forever.

Admin /admin

Token-protected extras: search-query leaderboard, zero-result searches, screen widths, traffic sources, geo demand. The token is the Worker's ADMIN_TOKEN secret.

Embed /embed

An iframe-ready map for the organizer's article, with copy-paste instructions at /embed. On phones the embed renders a map-only layout instead of redirecting. Embed traffic tags itself with ?src=embed so you can attribute it later.

7. Tracking, honestly

Every tracked interaction is a single POST of {action, label}, nothing else. No cookies, no user IDs, no precise location, and traffic from LAN/localhost hosts is dropped client-side so testing never pollutes the numbers. "Use my location" is opt-in and coarse (nearest neighborhood only). Counts are actions, not unique people.

Two lessons are baked in and worth remembering. Analytics Engine samples under load, so every query uses SUM(_sample_interval); a plain count silently ran ~10% low during Sandwich Week. And the Web Analytics RUM_SITE_TAG is not the beacon token from config.js; the discovery query lives in the wrangler.toml comment. Get it wrong and the audience numbers are silently empty.

8. Keeping repos in sync

Event repos share no git history with this template ("Use this template" squashes it), so sync is file-based, and the direction is always the same: build features here, push them outward.

The workflow when you build a new feature:

  1. Build and test it in this repo (python3 -m http.server 8000 --bind 127.0.0.1; skeleton data renders out of the box, or copy mock_data.js over a data-<year>.js).
  2. If you added a file, put it in template-manifest.json: files for shared code, themed if apply-theme brands it, nowhere if it's event content.
  3. Commit and push here (safe: nothing deploys from the template).
  4. In each event repo: python3 sync-from-template.py, review git diff, push.
Never patch template-owned code inside an event repo. If a live event forces a hotfix, fine, but port it to the template immediately after, or the next sync erases it. The tripwire is python3 sync-from-template.py --check: it exits nonzero on any drift.

If the feature needs a new config.js key, write the code to default sensibly when the key is missing, because config is event-owned and won't be updated by sync. Add the key to each event repo's config by hand.

9. The restaurant registry registry/

Venues repeat across events, and the painful parts of venue data (hand-captured Apple Maps links, verified coordinates) should be captured exactly once. The registry keeps every venue seen across all events, keyed by a stable slug, with multi-location support, name variants, contacts, and participation history. Participation history also generates each event's firstYearByName, so returning badges come for free.

CommandWhat it does
registry.py pull --roster f --year YRoster of names in, prefilled data-Y.js + data.js out, plus the firstYearByName block for config.
registry.py writeback --event e --year YMerges a concluded event's data file back in. Part of wind-down. Safe to re-run.
registry.py checkIntegrity: canonical areas, coordinates present, no name collisions.
registry.py seed --repos ...The one-time build from existing repos. Already done; you shouldn't need it again.
Merge policy worth knowing: newer events win field conflicts, but a curated share link (maps.app.goo.gl or maps.apple) is never overwritten by an auto-generated search-URL fallback.

10. Winding down & the report

One command, and the order matters enough that the script enforces it:

python3 scripts/wind-down.py

It archives hourly data and bakes the final tracking-snapshot.js before disabling anything (the raw analytics rows expire after roughly 90 days; the committed snapshots are what keep the stats dashboard alive forever), then sets archived: true, nulls the analytics beacon, and re-themes. The Worker and the scheduled Actions turn themselves off on their own schedule; trackUrl stays set so the stats pages keep reading historical aggregates.

Then generate the organizer report:

python3 scripts/generate_report.py

It builds a print-ready event report plus a one-pager per restaurant, entirely from the committed archives, into report/. Review before committing: on a live Pages domain, committing publishes it at /report/.

11. Good to know

The event ended but people can still thumbs-up. Bug?
No. Voting stays open for 5 days after the end date so late diners get their vote in, then closes on its own. The Worker stops recording a day after that.
Months later the stats page is empty. What happened?
The event was probably wound down without baking tracking-snapshot.js, and the raw analytics expired. The current wind-down.py bakes it automatically; for an already-expired event, bake from the newest committed daily snapshot in snapshots/.
Does the map track visitors?
Only anonymous action counts (see section 7). No identifiers, no precise location, and localhost/LAN traffic never counts.
How do I preview the pre-launch skeleton after launch?
Add ?year=9999 to the URL. And ?year=2026 forces a specific year's data file.
Can I edit app.js in an event repo just this once?
You can, and the next sync-from-template.py will erase it. Make the change in the template and sync it out; that's the whole system.
What does an archived event cost to keep running?
Effectively nothing. GitHub Pages is free, the Worker only serves cached reads, the Actions gate themselves off, and the Places API is only called inside the event window.
Is it safe to re-run the scripts?
Yes by design: apply-theme.py, sync-from-template.py, and registry.py writeback are idempotent, and wind-down.py refuses to run twice.
